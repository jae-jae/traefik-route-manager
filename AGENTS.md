# AGENTS.md

This file provides project-level working guidance for AI agents operating in the `traefik-route-manager` repository.

## Communication And Working Style

- Use English by default unless the user explicitly asks for another language.
- Keep responses direct, clear, and actionable.
- When you need library usage, framework behavior, or external integration details, prefer Context7 and DeepWiki for documentation lookup.
- Understand the existing implementation before changing code; do not rewrite behavior based on guesswork.

## Project Overview

- This is a database-free route manager for Traefik File Provider configurations.
- The backend is written in Go, and the frontend uses React 18 + TypeScript + Vite.
- Routes are stored as individual files in `CONFIG_DIR`, with the fixed `trm-` filename prefix.
- Only manage route files generated by this project; do not accidentally modify user-maintained Traefik config files.

## Key Directories

- `main.go`: application entrypoint.
- `config/`: environment variable loading.
- `internal/service/route_service.go`: core route file IO and YAML generation logic.
- `internal/handler/`: HTTP handlers.
- `internal/server/`: route registration and server startup.
- `internal/middleware/auth.go`: Bearer token authentication middleware.
- `internal/model/route.go`: route model and validation rules.
- `web/`: frontend project.
- `third_party/gin/`: local replacement for Gin; avoid editing it unless necessary.
- `SKILL.md`: API usage instructions for external AI assistants.

## Environment Variables

- `AUTH_TOKEN`: required, used for login and API authentication.
- `CONFIG_DIR`: required, points to the Traefik dynamic config directory.
- `ADDR`: optional, defaults to `0.0.0.0:8892`.

## Domain Rules

- Each domain maps to one config file named `trm-{domain}.yml`.
- Route listing should only read `trm-*.yml` files.
- When `https=true`, the primary router must use `websecure` and include `tls: {}`.
- When `redirectHttps=true`, generate an additional `web` entrypoint router and a `redirectScheme` middleware.
- Backend URLs must include a scheme, for example `http://192.168.1.10:3000`.
- If you change YAML generation logic, also verify that the read/parsing logic remains compatible.

## Change Guidance

- When changing backend handlers or validation rules, update or add tests in `internal/service/*_test.go` and `internal/server/*_test.go` first.
- When changing frontend form fields or API contracts, verify `web/src/api/client.ts`, `web/src/components/RouteForm.tsx`, and `web/src/components/RouteList.tsx` stay in sync.
- When changing authentication flow, confirm the login endpoint and Bearer token middleware still behave consistently.
- When changing user-facing capabilities, evaluate whether `README.md` and `SKILL.md` also need updates.

## Before Completion

- Run available verification commands instead of saying something “should work”.
- At minimum, choose verification based on the change scope:
  - Backend changes: `go test ./...`
  - Frontend changes: `cd web && bun run build`
- If verification is not run, explicitly say why.
